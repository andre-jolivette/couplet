# Couplet вЂ” Claude Code Reference

## On Genre and Assumptions

**Do not use genre labels as design constraints.** The developer's personal library happens to contain a particular style of photography, but Couplet is designed for any single-photographer library. Genre vocabulary ("street photography," "documentary") is likely to pull reasoning in the wrong direction: it suggests a narrower set of subjects, aesthetics, and pairing expectations than the tool actually supports, and it may lead to calibration decisions that are only correct for one kind of library. When assessing scoring, surfacing, or design questions, reason from what is actually in the data and what the pairing theory says вЂ” not from assumptions about what "street photographers" typically shoot or value. See decision #79.

## Project Overview
Couplet is a macOS app for photographers. It discovers meaningful image pairs in a photo library вЂ” not duplicates or sequential shots, but conceptually resonant connections the photographer might not have noticed.

## Architecture

### Indexing Pipeline (8 phases)
1. **Scan** вЂ” FileScanner reads EXIF captureDate, colorProfile from CGImageSource
2. **Duplicate detection** вЂ” dHash perceptual hashing, Hamming threshold=6; plus filename-variant rule: images whose filenames are numeric prefix/suffix export variants (`FilenameVariants.areVariants`) AND share an identical EXIF captureDate are unioned into the same duplicate group вЂ” catches crop/re-export copies that drift 7вЂ“20 Hamming bits past the dHash threshold (decision #94). Groups mark one hero (earliest captureDate, tie в†’ lowest id); non-heroes stay `isActive=1` but are excluded from pair scoring via `loadHeroVectors()` (`WHERE isHero = 1 AND isActive = 1`). Stale `duplicateGroups` rows are deleted each run before groups are re-created.
3. **Thumbnails** вЂ” 512px via CGImageSourceCreateThumbnailAtIndex (prevents IOSurface exhaustion)
4. **CLIP extraction** вЂ” CLIPCoreMLEngine, 224px input, cosine similarity embeddings
5. **Captioning** вЂ” OllamaCaptioningEngine в†’ `qwen2.5vl-caption` via localhost:11434; captions ALL uncaptioned active images each run. 768px input thumbnail (generated fresh вЂ” not the 512px cache), greedy decoding (temp 0, top_k 1), prompt v2: physical facts first, direction of action, emotion only from named visible evidence вЂ” decisions #50, #97
5.5. **Role extraction** вЂ” `OllamaRoleExtractionEngine` в†’ `qwen2.5:14b-instruct` via localhost:11434; extracts a per-image `RoleProfile` (JSON in `images.roleProfile`) from the caption for all active images WHERE roleProfile IS NULL. Text-only (reads the caption вЂ” no image decode), sequential, blocking; skipped when all present so normal re-indexes aren't slowed. Feeds the role-join entry-gate candidate generator (Phase 8.5). RoleProfile = subjects, phenomena (source/receiver), claims (sign text normalized to meaning), enacts/subverts (concepts the subject embodies/blocks), objects (+register real/toy/depicted/costume + category hypernym), directed_at, stance вЂ” decision #102
6. **Accent color extraction** вЂ” backfills `accentHue` / `accentSaturation` for all active images WHERE accentHue IS NULL; uses 256px downsample, 24 hue bins (15В°), prominence = area Г— mean saturation, 5вЂ“40% pixel fraction window, saturation floor 0.25; NULL for B&W/neutral images вЂ” decision #54
7. **Saliency centroid + gaze extraction** вЂ” backfills `weightCentroidX`, `weightCentroidY`, and `gazeDirectionX` for all active images WHERE weightCentroidX IS NULL OR gazeDirectionX IS NULL; three Vision requests in one pass: `VNDetectHumanRectanglesRequest` (primary centroid: area-weighted human bounding boxes, confidence в‰Ґ 0.3), `VNGenerateAttentionBasedSaliencyImageRequest` (centroid fallback for non-human subjects), `VNDetectFaceLandmarksRequest` (gaze: pupil offset within eye contour в†’ [-1=left, +1=right], no fallback вЂ” returns nil when pupils not detected); runs on cached 512px thumbnails вЂ” decisions #59, #64, #65, #69
8. **Pair scoring** вЂ” PairScorer, four-pool topK (composite top-150 + thematic top-50 + geometric top-5 + aesthetic top-5); two-phase: intra-folder first (blocking), cross-folder in background (cancellable) вЂ” decisions #34, #68, #75, #76
8.5. **Role-join candidate generation** вЂ” `IndexingEngine.generateRoleCandidates()` runs in the cross-folder task after scoring (decision #102). Loads all hero `RoleProfile`s, runs the deterministic `RoleJoins` rules (join 1 source/receiver gaze-excluded; join 2 claimв†”enact/subvert with claims un-gated; join 3 object real-vs-alt with category whitelist; joins 4/5 deferred), capped 4/image/join-type (~1,249 candidates). For each: NEW pairs are INSERTed (`selectedFor='role'`, axis scores via PairScorer); pairs that already exist are UPDATEd to set `roleHypothesis` (clearing any cold ThematicV2 verdict) so a composite/aesthetic pair with a genuine role connection still gets validate()d. The proposed connection is stored in `pairs.roleHypothesis` (v17) and fed to the judge. Surfaces Mode-1/2 third-meaning pairs the four-pool topK never reaches (backlog #95).
9. **ThematicV2 background pass** вЂ” `ThematicV2BackgroundPass` (actor, ConjunctEngine) runs after each page-0 grid load, judges candidate pairs sequentially via `ThematicScorerV2` using **qwen2.5:14b-instruct** on Ollama (LIMIT 750). **Role candidates first (decision #102):** pairs WHERE `roleHypothesis IS NOT NULL AND thematicV2Score IS NULL` are judged via `validate(captionA:captionB:hypothesis:)` вЂ” the judge *confirms or rejects the proposed connection* (it scores 6/7,4/4 this way vs 0/8 cold). **Non-role pool (decision #100/#95, fills remaining budget):** pure-color accent-echo pairs (`accent_echo AND thematicScore < 0.15 AND geometricScore < 0.30`) excluded; others deprioritised to the tail via `ORDER BY (accent_echo ? 1 : 0) ASC, compositeScore DESC`; judged via cold `score()`. Writes `thematicV2Score / thematicV2RelationshipType / thematicV2Rationale` to DB. Triggered from `EngineController.startThematicV2Pass()` after `streamPage0Pairs` completes; cancelled when a new index starts. **Near-duplicate guard:** candidates with identical `captureDate` (burst shots) excluded by SQL; filename export variants (via `FilenameVariants.areVariants`) excluded in Swift after the fetch. See decisions #82, #83, #94, #100, #102.

**Concurrency (decision #101):** Phases 1.5 (dHash compute), 3 (thumbnails), 4 (CLIP), 6 (accent), 7 (Vision saliency/gaze), and 8 (pair scoring) all run concurrently via the bounded `mapConcurrent(_:concurrency:work:consume:)` helper on `IndexingEngine` (Phase 4 CLIP uses its own equivalent task group). `work` is `@Sendable`/pure and runs on the task pool; `consume` (DB writes) runs serially on the engine actor. Caps auto-scale: `cpuConcurrency = processorCount` (1.5/3/6), `visionConcurrency = min(processorCount, 6)` (Phase 7), CLIP at `min(cores, 4)`. Pair-scoring parallelism is bit-faithful: `PairScorer.score` is pure, and the four-pool topK uses `orderedByScore(_:_:)` with a stable `(score, imageAID, imageBID)` tiebreaker so the stored set is reproducible regardless of completion order. Phase 1 file-scan colour-profile backfill and the Ollama phases (5 captioning, 9 ThematicV2) remain sequential. See decision #101.

### Three Scoring Axes (PairScorer.swift)
- **Aesthetic (weight 0.40)** вЂ” Three-way max: HSL histogram intersection (harmony), LAB palette contrast (contrast), or accent color echo (accent_echo). Echo score = `hueScore Г— в€љ(satA Г— satB)` where hueScore ramps в‰¤10В°в†’1.0, в‰¤30В°в†’linear, >30В°в†’0. Winning pathway sets `aestheticSubmode`. B&W pairs: harmony and contrast are multiplied by 0.65 вЂ” the 8-bin lightness-only comparison is less discriminative than the full 1,152-bin colour histogram, so scores are discounted rather than suppressed; a genuinely tonal-resonant B&W pair (both high-key, both moody) should score ~0.55вЂ“0.70; echo unaffected since `accentHue` is nil for B&W. Contrast normalization: `/100` (was `/80`) вЂ” avg LAB distance 100 = score 1.0. `colorProfile` must be fetched in `imageMeta` and passed to `PairScorer.score()` for suppression to fire. See decisions #56, #77.
- **Geometric (weight 0.20)** вЂ” three-component formula: `structuralГ—0.50 + directionalГ—0.25 + breathГ—0.25`. Structural = (edge orientation cosine Г— edgeMult + grid cosine Г— varMult) / 2; directional = `max(centroidScore, gazeScore)` вЂ” centroidScore from `directionalComplementScore()` using per-image `weightCentroidX/Y` (Vision human-detection centroid, decision #64); gazeScore from `gazeConversationScore()` using per-image `gazeDirectionX` (VNDetectFaceLandmarksRequest pupil offset, normalized by dividing by 0.30 to fill [-1,+1] before scoring, decision #70); gaze scored symmetrically вЂ” both directions tried, max taken вЂ” so pairs aren't missed due to arbitrary DB ID ordering (decision #71); rightward-gazer is always stored as imageAID (left display) for correct diptych orientation; orientationScore from `orientationOppositionScore()` using stored 32-bin edge histogram вЂ” dominant bin folded to undirected [0,16), scored as `sin(distГ—ПЂ/16) Г— в€љ(normPeakAГ—normPeakB)` with a hard diagonalness gate: both images must have dominant direction in the 22.5В°вЂ“67.5В° zone (`|sin(undirBinГ—ПЂ/8)| в‰Ґ 0.50`); bins within ~22.5В° of horizontal or vertical return 0 (decisions #73, #74); `directional = max(centroidScore, gazeScore, orientationScore)`; breath = `abs(normVarA в€’ normVarB)`. When directional > structural: `geometricSubmode` = `"gaze_conversation"` / `"opposing_diagonals"` / `"directional_complement"` based on which directional signal won. Rationale strings per submode. `geometricSubmode` stored per pair in DB (v13 migration). See decisions #59, #60, #64, #65, #67, #70, #71, #73, #74.
- **Thematic (weight 0.40, boosted to 0.60 when dominant)** вЂ” weighted Dice coefficient on ConceptClusters matched from qwen captions; CLIP cosine fallback when no captions. Boost fires when `thematic >= 0.20 && thematic >= max(aesthetic, geometric Г— 0.8)` вЂ” i.e., thematic must be the dominant axis. Before #75, the boost fired at `thematic >= 0.20` unconditionally, penalising strong two-axis pairs. See decision #75. **ThematicV2 override:** at display time, `effectiveThematic = thematicV2Score ?? thematicScore` вЂ” the LLM pair-level score replaces the cluster score when available. See decision #82.

## Pairing Theory вЂ” Design Intent and Known Gaps

Full theory in PAIRING_THEORY.md. This section is the operational summary for implementation work.

### Three Pairing Modes
- **Mode 1 вЂ” Semantic arc:** Two images occupy complementary positions in the same human experience arc. Neither image alone names it; together they make it visible. Third meaning is an idea that can be articulated. *Primary carrier: thematic axis.*
- **Mode 2 вЂ” Slant rhyme:** Two images share one specific formal property (a color, a shape, a quality of light) while diverging on everything else. Third meaning is a perception felt before it's named. *Primary carrier: aesthetic (accent echo) and geometric axes.*
- **Mode 3 вЂ” Ambient existential register:** Shared quality of attention to the fragile and ordinary (Soth's towels). Out of scope currently.

### Third Meaning Test
A pair passes if it creates a meaning that exists in neither image alone. It fails if the best description of the pair just restates what each image independently contains. Two dogs = fail ("there are two dogs"). Musician + ears-woman = pass (sound as a force in the city вЂ” given by one, hungered for by another).

### Per-Axis Design Intent vs. Current State

**Thematic** вЂ” *should measure:* complementary positions in the same human experience arc. *Currently measures:* weighted Dice on cluster vocabulary вЂ” rewards shared clusters, not relational position. *Architectural ceiling reached:* can't distinguish source from receiver of the same phenomenon. Musician and ears-woman both fire `sound_music` + `sensory_overwhelm` via the word "ear" в†’ Dice > ambient floor в†’ axis bonus guard fires в†’ pair below thematic topK cutoff. Caption redesign (#50) is the correct next lever. Do not add more clusters expecting this to improve.

**Aesthetic** вЂ” *should measure:* harmony (same visual world), complement (productive tonal contrast), or echo (one specific formal property rhyming across dissimilar images). *Currently measures:* three-way max of HSL histogram intersection (harmony), LAB palette contrast (complement), and accent hue echo (#56). *Missing:* light quality (deferred, hardest to compute reliably); tonal weight complementarity for breath pairs (#55, adds ~0.12 max composite lift when implemented).

**Geometric** вЂ” *should measure:* structural rhyme, directional complement (figures spatially facing each other / strong lines leading from one image to the next), or breath (dense image paired with spare/open image). *Currently measures:* three-component formula `structuralГ—0.50 + directionalГ—0.25 + breathГ—0.25` вЂ” structural via edge/grid cosine similarity, directional via `max(centroidScore, gazeScore)` where centroidScore uses Vision human-detection centroid opposition (`weightCentroidX/Y`, decision #64) and gazeScore uses face landmark pupil direction (`gazeDirectionX`, decision #65), breath via tonal weight differential. Geometric topK pool (top-5 per image with submode variety bonus) gives gaze_conversation and directional_complement pairs an escape hatch from composite dominance (#68). *Missing:* uniform area ratio for genuine sparseness detection; light quality.

### Known Scoring Failures
- **Musician + ears-woman:** Both captions share `sound_music` + `sensory_overwhelm` vocabulary в†’ Dice > ambient floor в†’ axis bonus guard fires. Pair scores via Dice but ranks below thematic topK cutoff. Only caption redesign (#50) fixes this.
- **Breath pairs:** Geometric differential ~0.07 composite max is below surfacing threshold. Aesthetic tonal weight complementarity (#55) needed to close the gap.
- **Mode 2 beyond color echo:** Light quality echo and gestural/energetic echo are unmeasured anywhere in the system.

## Known Gotchas

**Never run Ollama with a quantized KV cache (`OLLAMA_KV_CACHE_TYPE=q8_0`)** вЂ” it corrupts qwen2.5vl generation into systematic article/function-word garbling ("a aftermath", "He phone, held in his right hand, is a only object") even at greedy decoding. llama3.2 tolerates it, so the breakage is silent and model-specific вЂ” captions degrade while ThematicV2 keeps working. The Homebrew service plist (`/opt/homebrew/opt/ollama/homebrew.mxcl.ollama.plist`) had this set; `OLLAMA_FLASH_ATTENTION=1` alone is fine and can stay. If garbled articles reappear in captions, check `ps eww $(pgrep ollama) | tr ' ' '\n' | grep OLLAMA` first. See decision #97.

**B&W aesthetic discount requires colorProfile in imageMeta** вЂ” `PairScorer.aestheticScore()` discounts harmony and contrast scores (Г—0.65) for B&W pairs via `colorProfileA/B` params. These are fetched from the `images` table into the `imageMeta` tuple in both `runIntraFolderScoring` and `runCrossFolderScoring`. If either SQL SELECT or `imageMeta` typealias is modified, keep `colorProfile` in the query. Without it, B&W pairs will revert to inflated aesthetic scores. See decision #77.

**Filename-variant matching must be asymmetric вЂ” use FilenameVariants, never normalize-both-and-compare** вЂ” stripping `^\d+-` from both filenames silently fails when the base name itself starts with digits: `00-20250504-_R017085.jpg` loses only `00-`, but the original `20250504-_R017085.jpg` loses its *date* prefix, so the two normalized names never match. This bug made `sharesBaseName` (PairScorer) and `normalizedBaseName` (ThematicV2BackgroundPass) no-ops for this library's `NN-DATE-NAME` naming вЂ” crop-duplicate pairs entered the DB through the aesthetic topK pool with harmony в‰€ 0.94. All filename-variant checks now go through `FilenameVariants.areVariants()` (ConjunctEngine/Analysis), which compares each name's stripped forms against the *raw* other name. Do not reintroduce local normalize-both helpers. See decision #94.

**GRDB returns INTEGER columns as Int64, not Int** вЂ” `as? Int` silently returns nil on GRDB Row even on 64-bit macOS. Use the `intCol()` helper or explicit coercion: `(row["col"] as? Int) ?? (row["col"] as? Int64).map(Int.init) ?? 0`. Affects COUNT(*) results too.

**captureDate stored as INTEGER in SQLite** вЂ” GRDB's `as? Double` returns nil for Int64 column values. Explicit coercion required in `fetchRepresentativePairs` and `fetchPairs`: `(row["captureDateA"] as? Double) ?? (row["captureDateA"] as? Int64).map { Double($0) }`. Without this, temporal penalty never fires (all dates arrive as nil в†’ penalty=1.0). See decision #26.

**Titlebar (NSTitlebarBackgroundView)** вЂ” the frosted-glass effect comes from a private `NSTitlebarBackgroundView` at subview index 0 of `NSTitlebarView`, NOT from an NSVisualEffectView. The fix is `SolidTitlebarCover` (a plain CALayer-backed NSView) inserted at subview index 1 in `CoupletTheme.swift в†’ installSolidTitlebar`. Re-applies from `didBecomeKeyNotification` and `didBecomeMainNotification`. Read decision #30 in full before touching any titlebar rendering вЂ” prior approaches (targeting VEVs, setting layer.backgroundColor on NSThemeFrame) do not work.

**Tools bar (filter controls in titlebar)** вЂ” three approaches that do NOT work: (1) SwiftUI `.toolbar { ToolbarItem }` вЂ” macOS 15 applies per-item liquid glass capsules to NSToolbarItem containers that cannot be suppressed from inside; (2) `NSTitlebarAccessoryViewController` with `.bottom` вЂ” creates a visually separate second row with dead space between it and the traffic-light band; (3) `.windowToolbarStyle(.unified(showsTitle: true))` on the scene вЂ” causes the window title to reappear during state transitions. **What works:** empty `NSToolbar` (no items) with `window.toolbarStyle = .unified` expands `NSTitlebarView` to ~50px without producing NSToolbarItem containers; a `PassthroughHostingView<AnyView>` is inserted directly into `NSTitlebarView` at `leadingAnchor + 192` and updated each render cycle via `WindowConfigurator.updateNSView`. See decision #36 for full rationale and known limitations.

**weightedDice ambient floor** вЂ” `weightedDice()` in ConceptClusters requires в‰Ґ1 cluster in the shared intersection with weight в‰Ґ 0.75; pairs sharing only ambient-tier clusters (urban_street/nature_landscape/community_gathering/animal_presence, all weight 0.2) return `kAmbientFloor = 0.1`. Any new ambient cluster must stay at weight в‰¤ 0.24 or the gate logic breaks. See decision #29.

**Meaningful asymmetry gate requires weight в‰Ґ 0.75 unique clusters per side** вЂ” `PairScorer.swift` filters `onlyA` and `onlyB` to clusters with weight в‰Ґ 0.75 before checking asymmetry. A pair where one image uniquely has `urban_street` (weight 0.2) and the other has nothing unique at the meaningful tier fails the gate and returns ambient floor, even if both images have rich meaningful clusters they share. Changed in #49 from the original `!onlyA.isEmpty && !onlyB.isEmpty` which allowed ambient-only asymmetry to pass. Axis pairs are exempt from this gate вЂ” they fire via different clusters on each side by structural definition. See decision #49.

**Axis bonus only fires at ambient floor** вЂ” `ConceptClusters.axisPairs` defines 9 cluster-opposition relationships that reward complementary pairs (source в†” receiver of the same phenomenon). The bonus fires ONLY when `clusterScore в‰¤ 0.10` (ambient floor) вЂ” the guard `if saturated || clusterScore > 0.10 { axisBonus = 0 }` prevents +0.35 additive inflation on pairs that already score above ambient floor via Dice. A pair like musician + ears-cupping woman scores via Dice rather than the axis bonus if both captions happen to share `sound_music` or `sensory_overwhelm` vocabulary вЂ” the axis bonus won't additionally reward them. See decision #48.

**Temporal penalty must be replayed in convertToPair вЂ” two implementations must stay in sync** вЂ” `EngineController.convertToPair` (used in `fetchPairs` for the anchor strip) and `convertToPairFree` in `PairHelpers.swift` (used in the grid via `streamPage0Pairs`) are separate implementations of the same formula. Both must use: `в‰¤30s в†’ 0.40 | в‰¤60s в†’ 0.55 | в‰¤120s в†’ 0.75 | в‰¤300s в†’ 0.90 | >300s в†’ 1.0`. If you update one, update the other вЂ” divergence causes the anchor strip to show different composite scores than the grid for the same pairs. Decision #85 originally updated only `convertToPairFree`; decision #86 synced `convertToPair`. See decisions #26, #85, #86.

**Anchor strip DB fetch must not LIMIT below the image's pair count** вЂ” `EngineController.fetchPairs` with `anchorImageID != nil` uses `dbLimit = 500`. The `displayLimit` variable is only applied inside the `if anchorImageID == nil` branch, so the DB LIMIT is the sole display cap for anchor queries. Do not lower this below the expected hub-image pair count (407 in the test library); 500 gives headroom without hydrating an unbounded list. See decision #86.

**Orphan pair sweep runs before per-run DELETE** вЂ” `IndexingEngine.reindex()` sweeps pairs where either image `isActive=0` before the scoped per-run DELETE+INSERT. Preserve this ordering if touching `reindex()`. See decision #19.

**Two-phase scoring scope** вЂ” Phase 1 (blocking) scores batch Г— batch only; Phase 2 (background) scores batch Г— all-other-active. DELETE scoping is separate for each phase. `crossFolderTask` is cancelled before any new `index()` call. See decision #34.

**`mapConcurrent` parallel phases must never write to the DB from `work`** вЂ” the parallel indexing phases (1.5/3/6/7/8) run `work` on the concurrent task pool and `consume` serially on the engine actor. GRDB has a single writer: all `db.write` calls belong in `consume`, never in `work`. `work` must be pure/`Sendable` (no actor state, no DB). If you add a new parallel phase or move a DB write into `work`, you reintroduce concurrent-writer contention. The pure analysis funcs are static for this reason (`writeThumbnailFile`, `SaliencyAnalyser.analyse`, `ColourAnalyser.extractAccentColor`, `PerceptualHasher.dHash`, `PairScorer.score`). See decision #101.

**Parallel pair scoring requires the `orderedByScore` tiebreaker** вЂ” `PairScorer.score` is pure, so parallel evaluation is bit-faithful, but the four-pool topK selection uses unstable `sorted` + `prefix`. Without a deterministic tiebreaker the stored pair set would shuffle at *exact* score ties depending on parallel completion order. `orderedByScore(_:_:)` sorts by `(score desc, imageAID, imageBID)` in all four pools of BOTH scoring blocks (intra + cross-folder) вЂ” keep it on any new pool sort. (Note: a small run-to-run pair-count drift remains from Vision saliency/gaze ANE float nondeterminism вЂ” pre-existing, present on `main` too, unrelated to parallelism.) See decision #101.

**Double-onChange race in PairsGridView** вЂ” `LibraryViewModel.selectFolder` and `selectCollection` each set *both* `selectedFolderID` and `selectedCollectionID` in the same synchronous pass. If PairsGridView has separate `onChange(of: selectedFolderID)` and `onChange(of: selectedCollectionID)` handlers, both fire within milliseconds with different settled-state snapshots, launching two competing `loadPairs` calls. Fix: merge both handlers to call a single `reloadPairs()` helper that reads the *settled* `currentFolderID` / `currentCollectionID` published values (not the argument to `onChange`). Any future state mutations that touch both properties will hit the same race if separate observers are used. See decision #39.

**QueryService.fetchRepresentativePairs and fetchImagePairCounts are nonisolated** вЂ” both methods are `nonisolated` synchronous functions. Calling them directly from `@MainActor` context (e.g. inside `EngineController`) runs them synchronously on the main thread and blocks the UI for the duration of the SQL query. Always call them from a `Task.detached` block. The existing call site in `EngineController.fetchRepresentativePairs` already does this correctly вЂ” any new call sites must follow the same pattern. See decision #40.

**PairHelpers.swift functions must stay nonisolated** вЂ” `adjustedGeometricFree`, `convertToPairFree`, `pairSortComparator`, `applyCap2Free`, `applyPass2Free` are free functions in `PairHelpers.swift` marked `nonisolated` to prevent Swift 6.3 from inferring `@MainActor` on them. The inference chain is: `DisplayPair.colorA/colorB` use `NSColor` (which is `@MainActor` in the macOS 14+ SDK) в†’ Swift infers `@MainActor` on the `DisplayPair.init` в†’ infers it on any function that constructs a `DisplayPair`. The `nonisolated` keyword on the helpers and `nonisolated init` on `DisplayPair` together break this chain. Do not remove these annotations. See decision #41.

**streamPage0Pairs populates representativePairsCache on completion** вЂ” `EngineController.streamPage0Pairs` runs DB fetching and cap-2 in a `Task.detached`, yields accepted batches through `AsyncStream<[DisplayPair]>`, then updates `representativePairsCache` via `await MainActor.run` (with generation check) after the last batch. `PairsGridViewModel.loadPairs` consumes the stream and appends batches directly to `allPairs`. `loadMorePairs` still slices from the cache as before. **ThematicScorerV2.score() throws on connection error, returns nil on LLM parse failure** вЂ” `score()` is `async throws`. A throw indicates a network or HTTP error (Ollama unreachable, non-200 status) and the caller (`ThematicV2BackgroundPass.run()`) increments the consecutive-connection-failure counter toward the abort. A nil return means the LLM produced unparseable JSON (unescaped quotes in rationale when the model quotes sign/text content verbatim, or a truncated response). Nil resets the connection counter and skips the pair without aborting. Do not collapse these two paths back to a single `return nil` вЂ” that would cause parse failures to abort the pass after 3 consecutive malformed responses. See decision #89.

**startThematicV2Pass must set isThematicV2Running = true synchronously** вЂ” `isThematicV2Running = true` is assigned directly in `startThematicV2Pass()` (which runs on `@MainActor`) before `Task.detached` is called, NOT inside the task body. ## Workflow

When completing any feature or fix, follow these steps in order before considering the task done:

1. **Branch** вЂ” work on a branch named `[type]/[decision-id]-[short-description]` (e.g. `feat/54-accent-color`, `fix/26-temporal-penalty`). Create it at the start, not the end.
2. **Code complete** вЂ” implement and verify the change works.
3. **Docs** вЂ” update `CLAUDE.md` and `DECISIONS.md` to reflect the change. Check both files deliberately: not just the obvious section, but anything touched by the change (Known Gotchas, Open Backlog, Architecture overview).
4. **Commit** вЂ” use `#ID` prefix convention. Doc updates go in the same commit or a follow-up commit on the same branch. Do not leave doc updates uncommitted.
5. **Merge check** вЂ” confirm all commits are on the branch and nothing is dangling or uncommitted.
6. **PR ready** вЂ” confirm the branch is pushed. State the suggested PR title and a one-paragraph description summarising what changed and why.
7. **Branch cleanup** вЂ” after the PR is merged, delete the remote branch.

Do not report the task as complete until all 7 steps are done.
