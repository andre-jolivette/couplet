# Couplet — Architectural Decisions & Current State

> **What this file is:** A running log of every significant architectural decision made during development, in order. If you're trying to understand why something is built the way it is, start here. The [CLAUDE.md](CLAUDE.md) file has the current operational summary; individual decision entries in this log are the authoritative record of what was built and why.

## Project Brief
Couplet is a macOS app for photographers. It discovers meaningful image pairs in a photo library — not duplicates or sequential shots, but conceptually resonant connections the photographer might not have noticed.

---

## Architecture

Couplet is built as a sandboxed macOS app (Swift, macOS 14.0) with a Swift package (`ConjunctEngine`) handling all indexing, scoring, and database work, wired to a SwiftUI front-end via an `@MainActor` `EngineController` bridge. The core pipeline is an 8-phase indexing engine (scan → duplicates → thumbnails → CLIP → captioning → role extraction → accent/saliency → pair scoring) feeding a SQLite/GRDB database, with LLM background passes (`GazeVisionBackgroundPass`, `ThematicV2BackgroundPass`) that run after the grid loads. Three scoring axes — aesthetic (0.40), geometric (0.20), and thematic (0.40) — drive pair selection via a four-pool topK (composite top-150 + thematic top-50 + geometric top-5 + aesthetic top-5).

Current architecture is documented in [CLAUDE.md](CLAUDE.md). Individual decision entries in the log below are the authoritative record of what was built and why.

## Decisions Log

| Decision | Rationale |
|----------|-----------|
| Security-scoped bookmark for CLIP model | App Sandbox requires it; model lives outside app bundle |
| 512px thumbnail for CLIP input | Full-res caused IOSurface exhaustion during concurrent extraction |
| Soft temporal falloff (not hard cliff) | Hard 30s cliff was too aggressive for street photography shot quickly |
| Dice coefficient for thematic | Jaccard penalised protest photos (6 clusters) pairing with portraits (2 clusters) too harshly. Dice penalises each unique cluster once (denominator only) vs. Jaccard's double-penalty (numerator + denominator). Both asymmetry gate (≥1 unique per side) and saturation gate (>3 shared → zero) are preserved. Ceiling: 3 shared + 1 unique each → 6/8 = 0.75, comparable to aesthetic (0.84) and geometric (0.79) maxima in the current library. |
| 2000 DB fetch → cap → 750 display | Per-image cap + thematic re-weighting shrank 750 fetch to ~380 displayed |
| Per-image cap of 15 | Car interior image had 117 pairs, crowding out rest of library |
| topK=50 + thematic topK=10 | Thematic pairs never made composite top-50; separate selection ensures they appear |
| DELETE+reINSERT pairs on re-index | record.save() hit UNIQUE constraint silently; old scores were never updated |
| Caption all active images (not just scan batch) | Images from previous scans weren't getting captioned on re-index |
| colorProfile: pixel-level chroma sampling threshold=15 | `kCGImagePropertyColorModel == "Gray"` misses B&W JPEG exports from Lightroom/Capture One (stored as RGB, zero chroma). 32×32 downsample + max per-channel diff ≤ 15/255 catches these while absorbing JPEG ringing artifacts. |
| colorProfile backfill on every re-index | Migration default "color" affected all pre-v4 images. Running `refreshAllColorProfiles` after `upsertImages` (skipping the current scan batch) corrects the whole library from a single folder re-index. |
| Structured moondream prompt with Sentence 1/2 labels | Freeform prompts return empty string for ~90% of images; label scaffolding in instructions is required for reliable output even when asking moondream not to emit the labels. Retired — qwen2.5vl:7b (#20) uses a freeform paragraph prompt with no scaffolding. |
| Settings wired into EngineController at init | SettingsStore is @Observable (not ObservableObject); passed in at construction so fetchPairs can read live weights and threshold without async bridging |
| Thematic filter + live weight re-sort are display-time only | Scores stored in DB use default weights; re-sort on fetch avoids re-indexing on every settings change |
| TriangleWeightPicker takes three Binding<Double> params | Float↔Double bridging done at call site via computed Binding properties in SettingsView |
| Edge magnitude threshold raised 0.05 → 0.15 | Fine texture (brick, fabric, skin pores) was polluting the orientation histogram. Higher threshold filters these pixels before they vote, leaving only genuine structural edges. Baked-in constant in GeometricAnalyser — not a slider. |
| Geometric sub-scores stored per pair (v5) | rawEdgeSim, rawGridSim, maxEdgePeakedness, maxGridVariance persisted at score time. Enables display-time gating and re-weighting without re-score or re-index. Nil for pre-v5 pairs (fallback to stored geometricScore). |
| Display-time gating via two sliders | "Line direction strength" (edgePeakednessFloor, default 2.2) and "Composition structure floor" (gridVarianceFloor, default 0.12) apply binary discounts (40%/50%) at display time. Calibrated against library debug output: p10≈2.16 for peakedness, p10≈0.116 for variance. Floor discounts apply after the distinctiveness multiplier. |
| Distinctiveness multiplier (v6) | edgePeakednessMult = pow(normPeakA × normPeakB, 0.4); gridVarianceMult = pow(normVarA × normVarB, 0.4). Both images must be geometrically interesting for a pair to earn full credit — one flat image drags the score down proportionally. Normalization anchors at p90: kEdgePeakedNorm=4.0, kGridVarianceNorm=0.20. Exponent 0.4 (reduced from sqrt=0.5 — one calibration step toward 0 to soften the penalty). Multipliers stored per pair for display-time re-application. |
| Multiplier and floor gate are complementary, not alternatives | Multiplier: continuous, uses product of both images (both must be interesting). Floor gate: binary, uses max of both images (if even the stronger image doesn't clear the bar, discard). The multiplier fires first; the floor gate catches any remaining cases where the max is below the user's chosen threshold. |
| Orphan pair cleanup on every re-index | The per-run `DELETE FROM pairs WHERE imageAID IN (:ids) OR imageBID IN (:ids)` only covers images active in the current scan. Pairs for images deactivated via `removeFolder` were never removed, accumulating indefinitely (40,848 orphaned rows found — half the table). Added a sweep `DELETE FROM pairs WHERE imageAID NOT IN (active) OR imageBID NOT IN (active)` before the per-run DELETE. Runs unconditionally each re-index; cost is negligible on SQLite with indexed foreign keys. |
| Strip moondream label prefix at generation time | The structured prompt uses "Sentence 1:" scaffolding in its instructions; moondream (1.8B) sometimes echoes the label in output despite "No sentence labels" in rules. Previously: cleanup SQL nullified these captions, triggering a re-caption that often produced the same prefix again — an infinite cycle wasting ~152 ollama calls per run. Fix: (a) cleanup SQL now strips the prefix in-place using `TRIM(SUBSTR(caption, INSTR(caption, ': ') + 2))` instead of `SET caption = NULL`, preserving caption content; (b) `OllamaCaptioningEngine.caption()` strips the prefix from the raw response before storing. WHERE patterns unchanged (`'Sentence 1:%'` and `'First sentence:%'` only). Retired — both the cleanup SQL and the prefix-stripping code were removed when moondream was replaced by qwen2.5vl:7b (#20). |
| Weighted Dice for thematic scoring | All 24 clusters now carry a pairing weight (1.0 / 0.75 / 0.5). Score = 2×Σweights(A∩B) / (Σweights(A)+Σweights(B)). Tier 1.0: grief_sorrow, vulnerability_exposure, isolation_solitude, ritual_ceremony, tension_conflict, tenderness_care, devotion_belief, power_dominance, sensory_overwhelm. Tier 0.75: transformation_change, skilled_performance, labor_effort, stillness_rest, waiting_anticipation, movement_energy, bodily_gesture, looking_watching, confinement_freedom, youth_age, sound_music, joy_celebration. Tier 0.5: urban_street, nature_landscape, community_gathering. Weights calibrated against library frequency data (rationale field) — high-frequency ambient clusters scored lower. Known follow-on: a pair matching only a single 0.5-weight cluster still computes Dice=1.0 when neither image has any other clusters; a minimum weighted-shared floor is deferred to a subsequent session. |
| qwen2.5vl-caption: num_ctx capped at 2048; images resized to 512px before encoding | qwen2.5vl:7b defaults to a 128k context window. Ollama allocates KV cache proportional to num_ctx at load time — 128k costs ~2GB extra RAM and dominates inference time. Full-resolution camera files (12–24MP) encode to ~2,677 prompt tokens via qwen2.5vl's tiled patch encoder, which exceeds 2048 and causes degenerate output. Fix: OllamaCaptioningEngine now resizes every image to 512px using CGImageSourceCreateThumbnailAtIndex (same pipeline as CLIP) before base64-encoding. Measured token budget at 512px: ~247 vision + ~270 prompt + 250 response cap = ~767 total. num_ctx 2048 gives ~2.5× headroom. The Modelfile lives at `ConjunctEngine/Modelfile` and registers the model as `qwen2.5vl-caption`. Do not raise num_ctx casually — even 4096 doubles the KV cache cost for no benefit at these input sizes. |
| qwen2.5vl tokenizer special-token cleanup | qwen2.5vl:7b occasionally leaks tokenizer control tokens (`<|im_start|>`, `<|im_end|>`) into the generated response when generation is cut off at the num_predict boundary. `OllamaCaptioningEngine.caption()` truncates the response at the first occurrence of `<|im_` before storing. This is preferred over a post-hoc SQL cleanup — the DB never receives a malformed caption in the first place. |
| Captioning pipeline uses cached thumbnails | The captioning phase previously passed original image URLs to `OllamaCaptioningEngine`, which then re-decoded the full-res file (typically 12MB) to generate a 512px thumbnail on each call. Phase 2.5 (Thumbnails) runs before captioning and already writes 512px JPEGs to `~/Library/Containers/com.toastingmachine.Couplet/Data/Library/Caches/Conjunct/thumbnails/{imageID}.jpg`, recorded in `images.thumbnailPath`. The captioning SELECT was extended to fetch `thumbnailPath` and resolve it to a file URL; when the cached file exists it is passed directly to `caption()`, bypassing the re-decode. `thumbnailData(from:)` in `CaptioningEngine.swift` remains as a fallback for any image whose cache file is absent (e.g. OS cleared Caches). Measured saving: ~80ms per image (~90s across a 1,100-image library). |

| Weighted Dice ambient floor (kMinWeightedShared=0.75, kAmbientFloor=0.1) | A pair matching only a single tier-0.5 cluster (urban_street, nature_landscape, community_gathering) computes weightedShared=0.5, the only possible value below 0.75 in the discrete weight set {0.5, 0.75, 1.0}. Rather than returning 0, such pairs return kAmbientFloor=0.1 — a weak signal, not thematic resonance. The threshold 0.75 matches the lowest non-ambient tier weight so that a single bodily_gesture/looking_watching/etc. match still scores normally. | Five new clusters designed from photographer-influence research (#15) | Clusters designed by researching the work and aesthetic concerns of: Paul Graham, Rebecca Norris Webb, Daniel Arnold, Jeremy Paige, Melissa O'Shaughnessy, Andre Wagner, Rosalind Fox Solomon, Sage Sohier — the humanist street and documentary tradition. Real library images used to validate cluster boundaries and sharpen vocabulary during design. **`humor_absurdity`** (tier 0.75): strangeness that resolves into a punchline or visible delight. | Visual diversity multiplier on thematic score + expanded thematic topK (#45) | **Problem:** Cluster scoring rewards shared cluster volume. Same-event pairs (protest+protest, festival+festival) share many clusters → high Dice → dominate thematic topK-30. Cross-context resonant pairs (musician+ears, grief in different settings) share fewer clusters → lower Dice → excluded from topK entirely. Diagnostic confirmed: musician hero (2045) + ears hero (1523) are active, have captions, have feature vectors, but never made either image's composite topK-150 or thematic topK-30. **Root cause — topK density:** With 1,057 active images and topK=150, the 150th composite pair scores ~0.44–0.47. A cross-context pair with mismatched visual aesthetics scores below this threshold. The thematic topK-30 cutoff for the musician was ~0.46; musician+ears (estimated 0.37–0.42 thematic) fell outside. **Dead-end documented:** Caption embeddings (nomic-embed-text) tested as primary signal. Reverted: general-purpose text embeddings measure same-domain similarity, not cross-context resonance. Two festival images share festival vocabulary → high cosine → crowded out cross-context pairs further. **Fix 1 — Visual diversity multiplier (PairScorer.swift):** CLIP cosine is already computed at scoring time. Applied as a multiplier to `clusterScore` before the thematic-boost check: CLIP < 0.30 → ×1.35 (cross-context bonus); CLIP 0.30–0.60 → ×1.0 (neutral); CLIP 0.60–0.88 → ×0.75 (same-context penalty); CLIP > 0.88 handled by existing composite ceiling. **Effect:** protest+protest (CLIP ~0.65) drops from ~0.73 to ~0.55 thematic. Musician+ears (CLIP ~0.15–0.20) rises from ~0.38 to ~0.51. The multiplier applies to all pairs universally — no enumerated special cases. **Why better than complementarity bonus:** Does not require enumerating role relationships (sender/receiver). Any visually-distinct pair with cluster overlap gets rewarded regardless of the nature of the relationship. **Fix 2 — Thematic topK expansion (IndexingEngine.swift):** `thematicK = max(10, topK/5) = 30` → `max(30, topK/3) = 50` with topK=150. Applies to both Phase 1 and Phase 2 scoring. Combined with Fix 1, musician+ears thematic ~0.51 now fits within the top-50 thematic for the musician. **Pair count impact:** ~10–15% more pairs stored (~112K → ~130K expected). **Asymmetry gate confirmed not the culprit:** Ears has `isolation_solitude`/`sensory_overwhelm`/`grief_sorrow` that musician lacks; musician has `skilled_performance` that ears lacks. Both images have unique clusters; gate passes. Do not weaken or remove it. **Existing penalties preserved:** CLIP > 0.88 → composite ×0.40; CLIP > 0.75 && thematic < 0.20 → composite ×0.65. These remain unchanged — the diversity multiplier and composite ceiling are independent. |

| animal_presence demoted to ambient tier (#47) | **Diagnostic (2026-05-06, post-#45 re-index):** Dog+dog pairs ranked #7 globally (images 1447+1624, composite 0.623, thematic 0.547). Multiple other animal-mention pairs in the 0.58–0.62 composite range. The `animal_presence` cluster at weight 0.75 was the cause — it gave "two dogs" a meaningful thematic signal, exactly the same-same pattern the system is supposed to reduce. The visual diversity multiplier from #45 didn't help: two different dogs in different urban settings score CLIP cosine ~0.35–0.50 (neutral zone, ×1.0 multiplier). **Fix:** `animal_presence` weight changed from 0.75 → 0.2 in `ConceptClusters.swift`. At 0.2, it is now an ambient-tier cluster. Pairs sharing only animal_presence (and other ambient clusters) return the ambient floor (0.1) from `weightedDice` and score ≤ 0.1 thematic. Dog pairs that share a genuine emotional cluster (tenderness_care for a touching scene, isolation_solitude for a lonely stray) still score meaningfully via those clusters. The two-signal gate on `animal_presence` is preserved — it still requires both a specific animal subject noun AND a behavioral/relational context word to fire. This prevents incidental background animal mentions from contributing even the 0.2 ambient weight. **Why not remove the cluster entirely:** Removing it would cause animal pairs with no other cluster overlap to contribute nothing at all to the Dice denominator, inflating scores for pairs that DO share clusters. Keeping it at ambient weight preserves denominator contribution while preventing same-subject rewards. |

| Complementary axis pair bonus (#48) | **Problem:** The thematic Dice score rewards SHARED clusters — it measures "what do these images have in common?" Cross-register complementary pairs (one image is the source of a phenomenon, the other is the receiver) share NO clusters because they fire DIFFERENT clusters on each side. Musician fires `sound_music`, `skilled_performance`; ears fires `sensory_overwhelm`, `isolation_solitude`. Weighted Dice = ambient floor (0.1). After visual diversity multiplier (~1.35 for visually distinct images): thematic ≈ 0.135. Below the 0.20 boost threshold. The pair never enters DB. **Diagnostic confirmation:** Post-#45 re-index, musician (2045) + ears (1523) completely absent from the pairs table (123,326 rows total). Not low-ranked — absent. **Fix — axis pairs:** Added `ConceptClusters.axisPairs: [(a: String, b: String, bonus: Float)]` defining 9 cluster-opposition relationships: `sound_music↔sensory_overwhelm` (+0.35), `power_dominance↔vulnerability_exposure` (+0.35), `joy_celebration↔grief_sorrow` (+0.30), `skilled_performance↔looking_watching` (+0.25), `tenderness_care↔isolation_solitude` (+0.25), `power_dominance↔confinement_freedom` (+0.25), `labor_effort↔stillness_rest` (+0.20), `movement_energy↔stillness_rest` (+0.20), `devotion_belief↔tension_conflict` (+0.20). **Scoring integration (PairScorer.swift):** After computing `clusterScore` via Dice, compute `axisBonus = max(bonuses for any firing axis pair)`. The axis bonus fires ONLY when `clusterScore ≤ 0.10` (ambient floor) — the code guards `if saturated || clusterScore > 0.10 { axisBonus = 0 }`. This prevents additive inflation on top of a real Dice score (e.g. a pair that already scores 0.65 via Dice would reach T=1.35 capped at 1.0 without the guard). The saturation gate is also respected — same-event pairs (weightedSharedSum > 5.0) don't earn the bonus. The asymmetry gate is NOT required for axis pairs — the nature of axis pairs is that each image fires a DIFFERENT cluster, so asymmetry is structural. Apply: `thematic = min(1.0, (clusterScore + axisBonus) × diversityMult)`. **Post-implementation diagnostic (2026-05-06):** After re-index with changes #47, #48, and #49 active, musician (2045) + ears (1523) remains absent from the DB. Root cause: `"ear"` is present in BOTH `sound_music` and `sensory_overwhelm` keyword vocabularies (ears are both a musical instrument component and a sensory receiver). Additionally, `"intensity"` stems to `"intens"` which matches sensory_overwhelm. Both images therefore fire BOTH clusters and share them as a meaningful cluster intersection — Dice > ambient floor, so axisBonus = 0 (guard fires). The pair scores via Dice (~0.45–0.51 thematic) but ranks below the thematic topK-50 cutoff (~0.55–0.70) for this library. The axis pairs approach works correctly for pairs where images genuinely fire opposite-only clusters; it cannot help pairs where both images share both clusters due to overlapping vocabulary. Lesson: the keyword cluster system cannot distinguish "musician playing" from "listener overwhelmed" when both captions use the word "ear" or musical vocabulary. Caption prompt redesign is the recommended next step — see backlog #50. **Rationale string update:** When thematic ≥ 0.20 but no meaningful shared cluster exists, the rationale now checks axis pairs and emits "Complementary resonance — [A] meets [B]" instead of the generic "Conceptual contrast" fallback. Also fixed a pre-existing bug where the rationale picked `shared.first` which could surface ambient clusters ("urban street") — now filters to `meaningfulShared` (weight ≥ 0.75) before selecting the label. **Why axis pairs over role annotations or LLM scoring:** Axis pairs are enumerable, deterministic, and zero-cost at scoring time. Role annotations require per-cluster schema changes and still reduce to enumerating relationships. LLM scoring would require an inference call per candidate pair, expensive at 123K+ pairs. Axis pairs scale to any future relationship type by adding a single line to the array. |

| Meaningful asymmetry gate (#49) | **Problem:** The old asymmetry gate (`!onlyA.isEmpty && !onlyB.isEmpty`) was trivially satisfied by ambient-tier clusters. Two dog photos differing only by `urban_street` vs. `community_gathering` (both ambient, weight 0.2) passed the gate, allowing the Dice computation to proceed and producing inflated thematic scores via their shared tier-0.75 animal behavior clusters (`movement_energy`, `bodily_gesture`, `waiting_anticipation`). **Fix:** Changed the asymmetry gate in `PairScorer.swift` to require at least one meaningful cluster (weight ≥ 0.75) unique to each side: `let meaningfulOnlyA = onlyA.filter { (ConceptClusters.weights[$0] ?? 0) >= 0.75 }` and same for B; gate passes only when both are non-empty. Ambient-only asymmetry (one image has `urban_street`, the other doesn't) no longer counts as meaningful differentiation. **Effect:** Pairs where both images have identical meaningful clusters plus different ambient clusters no longer receive an asymmetry pass — they either score via shared meaningful Dice or return ambient floor. Pairs where each image has at least one unique meaningful cluster continue to score normally. **Interaction with axis pairs:** The axis bonus is not gated by the asymmetry check — when axis bonus fires, the pair is structurally asymmetric by definition (each image fires a DIFFERENT cluster on its side of the axis). **Calibration result:** Dog pairs that previously passed via ambient-only asymmetry now correctly return ambient floor or score only via shared tier-0.75 behavior clusters they genuinely share. |

| Caption embedding infrastructure removed (#51) | **What was built:** During backlog item #9 / #44, a caption-embedding pipeline was added to complement cluster scoring. Components: (1) `CaptionEmbeddingEngine.swift` — a `CaptionEmbeddingEngine` protocol plus `OllamaEmbeddingEngine` (POST `/api/embeddings` to nomic-embed-text, 768-dim L2-normalised float vectors) and `MockEmbeddingEngine`; (2) `v9_caption_embeddings` DB migration — `captionEmbedding BLOB` column on the images table (3,072 bytes/image); (3) Phase 3.6 in `IndexingEngine` — backfill loop for uncaptioned images; (4) `PairScorer.captionEmbeddingCosineSim` — raw cosine on float vectors, unused after revert; (5) `IndexingEngine` stores `embeddingEngine: any CaptionEmbeddingEngine` and fetched `captionEmbedding` in both intra- and cross-folder `imageMeta` queries; (6) `EngineController.buildEmbeddingEngine()` — availability check + engine construction, with `embeddingAvailable` published property; (7) `IndexingProgress.Phase.embedding` case. **What was tried:** nomic-embed-text cosines on qwen2.5vl captions live in [0.55, 0.95] not [0, 1].
